# Contributing

Thanks for your interest in contributing to BuffaloHerd OSS.

This organization hosts a mix of open-source software, research code, technical experiments, instructional material, templates, and supporting documentation.
Some repositories are stable projects; others may be exploratory, teaching-oriented, or works in progress.

Because of that, contribution workflows may differ from one repository to another.
Please read the README and any repository-specific contribution guidance before starting work.

## Ground Rules

- Be respectful and constructive.
- Keep changes focused and reasonably scoped.
- Follow the repository [Code of Conduct](CODE_OF_CONDUCT.md).
- Prefer clarity, maintainability, and useful documentation alongside code changes.

## Before You Contribute

Not every repository in this organization uses the same development workflow.

Before opening a pull request or making a substantial change, please check the repository for:

- project status
- canonical upstream location, if applicable
- issue tracker location
- branch and review workflow
- testing and development instructions
- any repo-specific contribution requirements

In some cases, GitHub may serve primarily as a public mirror or distribution point rather than the main location for development.

## Where to Ask Questions / Report Problems

Use the repository’s issue tracker unless the repository documentation directs you elsewhere.

If you are unsure whether GitHub is the correct place for a question, bug report, or proposed contribution, open an issue with a brief description and ask.

## Bug Reports

Please include, as appropriate:

- What you expected to happen
- What actually happened
- Steps to reproduce the problem
- Environment details (OS, language/runtime version, package versions, etc.)
- Logs, tracebacks, screenshots, or example input/output where relevant

## Suggestions / Feature Requests

Please include, as appropriate:

- The problem you are trying to solve
- Why it matters
- A proposed solution, if you have one
- Any relevant constraints or tradeoffs
- Links, references, or prior discussion if applicable

## Questions

If you are unsure whether something is a bug, a feature request, or a usage question, open an issue and describe what you are seeing.

## Contribution Workflow

Contribution workflows vary by repository.

For smaller changes, such as documentation fixes, typo corrections, or narrowly scoped improvements, a pull request may be appropriate directly on GitHub if the repository is accepting contributions here.

For larger changes, please open an issue first so the scope, direction, and appropriate contribution path can be discussed.

Where a repository provides more specific instructions, those instructions take precedence over this organization-wide guide.

## Style and Quality Expectations

Unless a repository states otherwise:

- keep changes focused and well scoped
- update documentation when behavior changes
- add or update tests when appropriate
- avoid unnecessary breaking changes
- follow existing project structure and coding style
- include clear commit and pull request descriptions

## Licensing

By contributing, you agree that your contributions will be licensed under the license used by the repository to which you contribute, unless that repository states otherwise.
